Vault becomes the single point of issuance, audit, and revocation for AAP tokens.

A Vault secrets engine that dynamically mints and revokes Ansible Automation Platform OAuth2 tokens. Each read of a credentials path mints a fresh token through AAP's REST API, hands it to the caller under a Vault lease, and deletes it from AAP when the lease ends — so no long-lived AAP token is ever copied into a pipeline, a tfvars file, or a CI secret store.

01 · What it is

A small engine that closes a real gap in AAP credential hygiene.

AAP OAuth2 tokens are long-lived by default — a fresh token carries a ~1-year server-side expiry — and AAP offers no built-in lease, audit, or just-in-time issuance story. Today those tokens get minted by hand and pasted into automation. This engine puts Vault in front of that flow.

For the caller

Read a path, get a token

A pipeline reads aap/creds/<role> and receives a working AAP token plus a lease. No token is stored anywhere durable; when the job ends the lease is revoked and the token disappears from AAP.

For the platform team

One control plane

Issuance, TTLs, scope, and per-identity ownership are defined as Vault roles. Vault's audit log becomes the record of who obtained AAP access and when — and a single lease revoke kills it everywhere.

For AAP

Native, reversible units

The engine creates only what AAP makes cheap and revocable: OAuth2 tokens against identities that already exist. It never mutates users, RBAC, or org structure — its blast radius on AAP is exactly "tokens it minted".

There is no official AAP Go SDK, so the engine calls AAP's REST API directly with net/http — the same approach Ansible's own terraform-provider-aap takes.

02 · The design decision

Why mint tokens, not dynamic users?

The reflexive instinct for a dynamic-secrets engine is "create a fresh principal per lease", the way the database and LDAP engines do. For AAP that's the wrong call — and it's worth refuting properly, because the reasoning is what makes this engine principled rather than improvised.

The deciding question What is the cheap, native, revocable "ephemeral unit" in the target system? A dynamic-secrets engine should create whatever the target makes plentiful and side-effect-free to throw away — no more, no less.
EngineEphemeral unit it createsWhy that unit
Database / LDAP a brand-new user per lease
CREATE USER · creation_ldif		 In a SQL DB or directory a user is a cheap row/entry. Creating and dropping them is trivial and side-effect-free.
AWS dynamic IAM user or STS token Both are first-class, cheap, API-managed objects.
Terraform Cloud / Enterprise an API token for an existing user / team / org TFC identities are heavyweight platform principals with rich RBAC. The cheap, plentiful, revocable thing is the token, not the principal.
AAP — this engine an OAuth2 token for an existing identity Same shape as TFC/TFE. AAP sits squarely in that row — and that is not a coincidence; they are the same kind of system.

AAP is TFC-shaped, not LDAP-shaped

  • AAP users are heavy platform principals. Tied to the gateway's auth system, often backed by external SSO/LDAP/SAML, may consume a license seat, and are useless until granted org/team membership and object-level RBAC. Churning them per-lease is invasive in a way dropping a Postgres role never is.
  • A fresh AAP user has zero permissions. To make a dynamic user useful you'd have to re-implement AAP's RBAC assignment inside the Vault role — encode "create user → POST role assignments → add team membership → …" as creation templates, then reliably tear it all down on revoke. That is rebuilding AAP's authorization model inside Vault config: high complexity, high blast radius if revoke half-fails.
  • AAP already exposes the right unit. It supports many tokens per identity, programmatic creation, and programmatic deletion (DELETE /tokens/{id}/) — exactly the "plentiful + revocable" property that makes the token the correct dynamic unit.
  • It matches how AAP is actually operated. Purpose-built service accounts (svc-galaxy-sync, aap_operator, …) carry the right RBAC; you issue tokens against them.
Litmus test Would creating the principal force you to also script its permissions? For a DB user you write one GRANT. For an AAP user you'd orchestrate org/team/object RBAC across several endpoints — a sign the principal is the wrong granularity to be ephemeral. The credential (token) is the right one.
Precedent HashiCorp made exactly this call for TFC/TFE: a rich-RBAC SaaS → mint tokens for existing principals, don't create principals. AAP is the same archetype, so copying TFC's shape is the principled choice, not a shortcut.

Decision-framework outcome: AAP supports multiple tokens + programmatic deletion → dynamic secrets (not dynamic users, not static). No existing Vault engine fits (not PKI / transit / identity) → a custom engine is justified.

03 · How it behaves, end to end

The full lifecycle of one dynamic token.

A single read of aap/creds/<role> drives the sequence below. The unusual parts — a write-ahead log before the mint, a read-back guard after it, and a credential snapshot into the lease — exist to make two failure modes impossible: handing back a mis-owned token, and leaking a token AAP created but Vault never leased.

request response internal step (Vault storage) guard / revoke

Figure 1 — one creds/ read, from request to revocation. The WAL brackets the mint; the guard brackets the response.

before the mint

Write-ahead log

The token's request marker is recorded in a Vault WAL before calling AAP. If the process dies mid-flight, the periodic rollback finds and revokes whatever AAP created.

after the mint

Read-back guard

For per-user and application-scoped roles the engine re-reads the new token's owner / binding and revokes + errors on any mismatch — a mis-owned secret can never reach the caller.

at lease time

Snapshot into the lease

The token id (as a string) and the AAP connection are snapshotted into the lease's internal data, so revocation works even after config is later changed or deleted.

04 · Architecture

Three actors, one privileged credential, two storage realms.

The engine is a thin, stateless-by-design translator between Vault's lease machinery and AAP's token API. The one subtlety worth a reviewer's attention is where the privileged credential lives — and why some copies of it cannot be seal-wrapped.

05 · Domain mapping

Everything is one of three Vault objects.

The engine's whole surface area maps onto Vault's standard config / role / creds triad — nothing exotic for an operator to learn.

config

The AAP connection

address, one auth scheme (token or username+password), tokens_api_path, TLS trust, request_timeout, description prefix, and automated-rotation fields. Writing it verifies connectivity before persisting.

role/<name>

The issuance policy

scope (read|write, default read), description, optional username+bootstrap_token for per-user ownership, optional application binding, and ttl/max_ttl.

creds/<name>

The leased token

A read mints a fresh AAP token under that role and returns token, token_id, scope, and expires beneath a renewable Vault lease.

Configure & define a role

# Point the engine at AAP (bearer is recommended) vault write aap/config \ address="https://aap.example.com" \ token="<privileged AAP token>" \ tokens_api_path="/api/gateway/v1" # An issuance policy: read-scope, 1h lease, 8h ceiling vault write aap/role/ci \ scope=read description="vault-ci" \ ttl=1h max_ttl=8h

Mint, renew, revoke

# Mint — returns a token under a renewable lease vault read aap/creds/ci # key value # lease_id aap/creds/ci/<id> # token <the AAP OAuth2 token> # token_id 31 # scope read # Extend the lease — same AAP token, no re-mint vault lease renew aap/creds/ci/<id> # Revoke early — also DELETEs the token in AAP vault lease revoke aap/creds/ci/<id>

06 · Engineering depth

The parts a secrets-engine reviewer will scrutinise.

These are the decisions where a token engine is easy to get subtly wrong. Each is a deliberate choice with a stated failure mode it closes.

mint → read back → revoke-on-mismatch

The guard pattern

Per-user and application-scoped roles can't trust AAP to honour a requested field. So after minting, the engine re-reads the token's actual owner / application and, on any mismatch, revokes the token and errors. The bad outcome — a secret with the wrong identity or binding — is made impossible, so the feature is safe even against an AAP version that silently ignores the field.

bootstrap_token + username

Per-user issuance on the 2.5 gateway

AAP assigns a minted token's owner to whoever authenticates the call — the user field is silently ignored, and there's no admin "mint-for-user" path. So to issue a token owned by svc-deploy, the role carries that account's own token as bootstrap_token and the engine authenticates as them. username is the guard that proves it worked. No user passwords stored; bootstrap tokens are seal-wrapped and never returned.

WAL · min-age 5m

No orphaned tokens

Minting is bracketed by a write-ahead log: the request marker is recorded before the AAP call and deleted on success. If Vault fails after AAP creates the token but before the lease is durable, walRollback revokes the orphan. One sub-second window (crash between CreateToken returning and PutWAL) is irreducible — AAP has no reserve-then-commit API — and is documented rather than hidden.

honest tradeoff

Revocation snapshot & blast radius

So a token can always be revoked even after config changes, the engine snapshots the AAP connection — including the privileged credential — into each lease. Engine storage (config, role/*, WAL) is seal-wrapped, but these lease copies live in core's expiration-manager storage, which a plugin cannot seal-wrap (barrier-encrypted only). Stated plainly: treat the privileged credential as present wherever active leases are, and rotate it through rotate-root, never out of band.

idempotent · precise

Revocation that converges

Revoke reads token_id from the lease and DELETEs it; a 404 counts as success, so Vault's retries converge. The id is stored as a string so it survives the lease's JSON round-trip exactly — a numeric id would decode back as float64 and lose precision above 253. Revocation also falls back to the current config credential if the snapshot was itself rotated away.

strategy A · renewable

Leases are the binding clock

AAP has no extend-expiry API, but its tokens default to a ~1-year expiry — comfortably longer than any sane max_ttl — so renewing the lease (same token) is sufficient, matching every first-party Vault engine. Earlier research proposed non-renewable leases; the live 1-year finding removed the drift risk that motivated it. rotate-root rotates the engine's own privileged token in place, serialized so overlapping calls can't strand bookkeeping.

One principle behind both guards Mint → read back → revoke-and-error on mismatch. The failure mode (a mis-owned or mis-scoped secret) is made impossible rather than merely unlikely — which is what lets per-user and application binding be safe even on an AAP version that ignores the requested field.
